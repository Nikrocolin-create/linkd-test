# План на лето: middle → senior

**Срок:** ~10 недель · **Бюджет:** 2–4 ч/день · **Диагностика:** 23/24 hard mode → концепции есть, нужен build + depth.

---

## Главная идея

Не читать ещё больше теории, а **построить одну production-grade систему**, в которой каждая из шести тем всплывает на senior-глубине. Проект — это спайн (spine). Вокруг него — ежедневные depth-репетиции по ротации и интервальные повторения.

```
        ┌─────────────── СПАЙН-ПРОЕКТ (каждый день) ───────────────┐
        │  строишь слой за слоем, ранние слои остаются живыми        │
        └───────────────────────────┬──────────────────────────────┘
                                     │
   ┌─────────────────┬──────────────┼──────────────┬─────────────────┐
 Java/Concurrency   JVM          Spring          DB/Mongo      System Design   k8s/DevOps
   └──────────── depth-репы по ротации (открытые вопросы, не MCQ) ────────────┘
                                     │
                     интервальные повторения: 1д → 3д → 7д → 21д
```

Почему так закрывает твою проблему «забуду первый месяц»: кэш, который ты написал на W2, ты **нагружаешь и тюнишь на W9**. Outbox с W5 ломается под нагрузкой на W9. Темы не уходят в архив — они переиспользуются.

---

## Спайн-проект: `linkd` — распределённый short-link + аналитика кликов

Ты уже делал design-walkthrough URL-шортенера. Теперь строим его по-настоящему и навешиваем все senior-заботы. Один проект бьёт по всем шести категориям:

| Слой | Категория | Что реально делаешь |
|---|---|---|
| Core API + Postgres | Java / Spring / DB | hexagonal-структура, constructor injection, @ConfigurationProperties, Testcontainers |
| Redis-кэш | System Design / Spring | cache-aside, TTL, защита от stampede (request coalescing) |
| ID-генерация + идемпотентность | System Design / DB | base62, collision handling, idempotency-key, стратегия шардинга |
| Rate limit + resilience | Concurrency / Sys Design | token bucket (Bucket4j/руками), circuit breaker (Resilience4j), тюнинг пулов |
| Outbox → Kafka → consumer | Sys Design / Java / DB | outbox pattern, идемпотентный консьюмер, Mongo для аналитики |
| Аналитика | DB (Mongo) | aggregation pipeline, $lookup, ESR-индексы, CQRS-lite |
| Observability | DevOps / Spring | Micrometer + Prometheus, structured logs, OpenTelemetry tracing |
| Kubernetes | DevOps | Deployment/Service/ConfigMap/Secret, probes, requests/limits, HPA, rolling update |
| Load test + harden | ВСЁ | k6/Gatling, поиск бутылочного горла, профилирование JVM, фикс N+1 |

> Проект можно заменить (event-driven order/payment с CDC через Debezium — тоже хороший кандидат), но `linkd` даёт максимальное покрытие при минимальном стартовом трении, потому что архитектуру ты уже знаешь.

## Расписание по неделям

Каждая неделя = **один primary build-фокус** (двигает проект) + **depth-репы по другим темам** (держат всё тёплым).

| Неделя | Build (primary) | Depth-репы (ротация) |
|---|---|---|
| **1** | Скелет: API create/resolve, Postgres, чистая структура, Testcontainers | DB-индекс под lookup; 1 concurrency-задача/день |
| **2** | Кэш-слой: Redis cache-aside, TTL, stampede/coalescing | уровни изоляции SQL, JMM-вопросы |
| **3** | ID-генерация, идемпотентное создание, стратегия шардинга/партиционирования | JVM-память, Spring-транзакции |
| **4** | Rate limiting (token bucket) + circuit breaker, тюнинг thread pool | k8s-основы, DB-блокировки |
| **5** | Outbox pattern → Kafka → идемпотентный consumer | JVM, Spring-прокси/AOP |
| **6** | Аналитика на Mongo: aggregation pipeline, ESR-индексы, read-модель | concurrency, k8s |
| **7** | Observability: метрики, трейсинг, structured logging | Sys Design, JVM **(DevOps добирает вес — твой единственный пробел)** |
| **8** | Kubernetes: deploy в kind/minikube, probes, HPA, rolling update | лёгкие репы по всем |
| **9** | Load test + harden: бутылочное горло, профилирование, фикс N+1, GC/off-heap | всё конвергирует под нагрузкой |
| **10** | Design-doc + mock-интервью (со мной) + полировка + буфер | retro, слабые места |

## Дневной шаблон (2–4 ч)

```
0:00–0:05   План дня: 1 build-задача + 2 depth-промпта
0:05–1:30   BUILD-блок (проект)          ← здесь живёт дневной минимум
1:30–1:40   перерыв
1:40–2:15   DEPTH-репы (открытые, вслух/письменно, из тем вне недели)
2:15–2:45   (хорошие дни) чтение DDIA/исходники/доки + 1 строка в лог
```

**Depth-репы — открытые, не MCQ.** Раз MCQ ты упёрся в потолок, реальные пробелы вскрываются только так:
- «Объясни вслух за 3 минуты, как работает X» (если поплыл — это пробел)
- «Что выведет этот код / где здесь баг»
- «Спроектируй Y на салфетке за 15 минут»
- «Здесь течёт память — почему»

## Система против «2 дня работаю — 3 дня ямлюсь»

Это не про силу воли. Срыв происходит, потому что один пропущенный день превращается в пропущенную неделю — холодный старт слишком дорогой. Механика:

1. **Запрет нулевого дня.** Минимум — 25 минут build-блока. Каждый день. Это пол, а не потолок: настолько мало, что не пропустишь, и обычно затягивает дальше.
2. **Останавливайся на середине.** Заканчивай сессию посреди задачи, оставив `// TODO: next` — назавтра рестарт тривиален (эффект Зейгарник). Это убивает холодный старт, который и роняет тебя на 5 дней.
3. **Стрик-лог.** Одна строка в день: что сделал. Цепочку не рвёшь.
4. **Тяжёлое — первым.** Самый сложный кусок в начале сессии, пока есть воля.
5. **Осязаемое > абстрактное.** Коммить что-то почти каждый день. Проект даёт дофамин, которого «учил system design» не даёт — ты сам это заметил.
6. **Пятничное демо.** Запусти систему, напиши 3 предложения: что работает, что дальше.
7. **Честность по энергии.** Плохой день → механика (тесты, рефакторинг, доки). Хороший → новая тяжёлая концепция. И то и другое = не нулевой день.

> Отдых — часть плана, а не провал. Один осознанный выходной в неделю запланирован. Срыв — это незапланированные 5 дней, а не один.

## Интервальные повторения (против «забыл synchronized»)

Маленькая колода depth-промптов по категориям. Промпт, который сегодня сделал, возвращается через **1 → 3 → 7 → 21 день**. Ошибся — интервал сбрасывается. Темы вне текущей недели держатся тёплыми именно так.

## Как используем чат

- **Экзаменатор:** кидаю открытые вопросы / «найди баг» / «спроектируй» — то, что нельзя обмануть форматом.
- **Code review:** скидываешь PR/кусок проекта — разбираю как senior-ревьюер.
- **Design review:** показываешь архитектурное решение недели — бью по слабым местам.
- **Колода:** веду твои depth-промпты и напоминаю по интервалам.
